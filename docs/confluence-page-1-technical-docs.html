1. Project Overview

Nordlys Home is a React 18 single-page application (SPA) built as a demo and showcase frontend for a fictional Scandinavian home furnishings retail brand. The application presents a complete e-commerce storefront experience including product browsing, category navigation, a shopping bag interaction, and a responsive layout across desktop and mobile viewports.

The project is intended as a frontend-only demonstration and currently contains no backend integration. All product and category data is defined statically within the application source code.

4. Architecture

4.1 Application Architecture

Nordlys Home is a pure client-side SPA. There is no server-side rendering (SSR), no API integration, and no build-time static site generation. The full application lifecycle is:

  1. The browser loads index.html from a static file host.
  2. Vite-bundled JavaScript assets are fetched and executed.
  3. React mounts the App component into the #root DOM node via main.jsx.
  4. All UI state is managed in-memory via React useState hooks.
  5. Navigation between visual sections is CSS-based scrolling — there is no client-side router.

6. Data Models

All data is defined as inline JavaScript arrays and objects within App.jsx. There is no external data source or schema enforcement.

6.1 Product

// Implied shape of a product object
{
  id: String,          // e.g. "linn-chair"
  name: String,        // e.g. "LINN Chair"
  price: Number,       // e.g. 89
  image: String,       // SVG data URI (inline mock image)
  category: String     // e.g. "Seating"
}

6.2 Category

// Implied shape of a category object
{
  id: String,          // e.g. "living-room"
  label: String,       // e.g. "Living Room"
  image: String        // SVG data URI
}

6.3 Perk

// Implied shape of a perk object
{
  icon: String,        // SVG or unicode icon
  title: String,       // e.g. "Delivery from $9"
  description: String  // Supporting copy
}

7. State Management

State is managed exclusively with React's built-in useState hook. There is no external state library (no Redux, Zustand, or Context API).

State VariableTypeInitial ValuePurpose
bagCountNumber0Tracks the number of items added to the shopping bag; displayed in the nav badge
lastAddedString | nullnullStores the name of the most recently added product; used to display a transient toast/notification
 
const [bagCount, setBagCount] = useState(0);
const [lastAdded, setLastAdded] = useState(null);

State update flow: When a user clicks "Add to Bag" on a product card, setBagCount increments the counter and setLastAdded stores the product name. A visual notification is shown and then cleared after a timeout.

8. Styling Conventions

8.1 Approach

All styles are defined in a single global styles.css file (509 lines). No CSS modules, styled-components, or CSS-in-JS libraries are used.

8.2 CSS Custom Properties

Design tokens are defined as CSS custom properties on :root for consistent theming:

:root {
  --color-brand: /* primary brand color */;
  --color-bg:    /* background */;
  --color-text:  /* primary text */;
  --radius:      /* border radius token */;
  --shadow:      /* box shadow token */;
}

10. Build and Deployment

10.1 Build Output

Running npm run build produces a /dist directory containing:

  • index.html — HTML entry point with hashed asset references
  • assets/ — Hashed JS and CSS bundles (e.g. index-[hash].js, index-[hash].css)

10.2 Deployment

Because the output is a collection of purely static files, the /dist folder can be deployed to any static hosting provider:

  • Firebase Hosting
  • Google Cloud Storage + Cloud CDN
  • Netlify / Vercel
  • AWS S3 + CloudFront
  • GitHub Pages

Info: No server-side runtime is required. The hosting platform only needs to serve static files and redirect all paths to index.html for SPA routing (not currently applicable since there is no client-side router, but required if routing is added in future).

10.3 Environment Variables

No environment variables are currently used. If a backend API is integrated in future, a VITE_API_BASE_URL variable should be introduced via a .env file and accessed as import.meta.env.VITE_API_BASE_URL.

11. Known Limitations and Future Improvements

11.1 Current Limitations

AreaLimitation
DataAll product and category data is hardcoded inline in App.jsx
RoutingNo client-side router — application is a single scrollable page
BackendNo API integration — cart interactions are UI-only and non-persistent
AuthNo user authentication or session management
AccessibilitySemantic HTML and ARIA roles have not been audited
TestingNo unit, integration, or e2e tests are present
TypeScriptNo type safety — all code is plain JavaScript
Component StructureAll UI is in a single 248-line App.jsx — no decomposition

11.2 Recommended Future Improvements

  1. Decompose App.jsx into individual feature components (Header, ProductGrid, CategoryGrid, Footer, etc.) in a src/components/ directory.
  2. Introduce React Router (v6) for proper URL-based navigation between product detail pages, cart, and checkout.
  3. Replace inline data with API calls to a real or mocked backend — use React Query or SWR for data fetching and caching.
  4. Add TypeScript — define interfaces for Product, Category, CartItem, and component props.
  5. Implement a cart store using Zustand or React Context + useReducer for persistent, structured cart state.
  6. Add a test suite — Vitest + React Testing Library for unit and integration tests; Playwright for e2e.
  7. Audit accessibility — ensure all interactive elements have appropriate ARIA labels, keyboard navigation, and focus management.
  8. Integrate a design token system — formalise the existing CSS custom properties into a structured token file.
