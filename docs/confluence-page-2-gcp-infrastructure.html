Nordlys Home — GCP Target Infrastructure

Document Type: Infrastructure Design  |  Project: Nordlys Home Retail Platform (Frontend + Backend)  |  Last Updated: 2026-02-24  |  Status: Proposed Architecture

Covers frontend SPA, Node.js/Express REST API backend, relational and document databases, CI/CD, security, and observability on Google Cloud Platform.

Warning: This document describes a target infrastructure design for a future state of the Nordlys Home application that includes a Node.js/Express REST API backend. The current codebase is frontend-only. This page should be used as a planning and engineering reference for the backend integration phase.

1. Architecture Overview

1.1 High-Level Design Principles

  • Serverless-first — Cloud Run for the backend eliminates server management overhead and scales to zero in non-production environments.
  • Separation of concerns — Frontend (static assets) and backend (API) are independently deployed and versioned.
  • Managed services — Cloud SQL, Firestore, and Secret Manager reduce operational burden.
  • Security by default — VPC isolation, Cloud Armor WAF, IAM least-privilege, and Secret Manager for all credentials.
  • Observability built in — Cloud Monitoring, Cloud Logging, and Cloud Trace from day one.

3.3 SSL / TLS

  • Google-managed SSL certificates are provisioned automatically on the Load Balancer.
  • All HTTP traffic is redirected to HTTPS via a redirect rule on port 80.
  • TLS 1.2 minimum enforced via SSL policy.

3.4 Cloud Armor Security Policy

# Create Cloud Armor policy with OWASP rules
gcloud compute security-policies create nordlys-armor-policy \
  --description "WAF policy for Nordlys Home"

# Enable OWASP ModSecurity CRS preconfigured rules
gcloud compute security-policies rules create 1000 \
  --security-policy nordlys-armor-policy \
  --expression "evaluatePreconfiguredExpr('xss-stable')" \
  --action deny-403

gcloud compute security-policies rules create 1001 \
  --security-policy nordlys-armor-policy \
  --expression "evaluatePreconfiguredExpr('sqli-stable')" \
  --action deny-403

# Rate limiting — 100 requests per minute per IP
gcloud compute security-policies rules create 2000 \
  --security-policy nordlys-armor-policy \
  --expression "true" \
  --action rate-based-ban \
  --rate-limit-threshold-count 100 \
  --rate-limit-threshold-interval-sec 60

4.2 Cloud CDN Configuration

  • CDN enabled on the Load Balancer backend service pointing to the Cloud Storage bucket.
  • Cache mode: CACHE_ALL_STATIC — automatically caches JS, CSS, fonts, images.
  • index.html is excluded from CDN cache via Cache-Control: no-cache header to ensure users always receive the latest deployment.

4.3 Frontend Deployment Script

# Upload assets with long cache (hashed filenames)

6. Database Design

6.1 Design Philosophy

The database architecture uses a polyglot persistence pattern:

  • Cloud SQL (PostgreSQL) — for structured, transactional, relational data where consistency and ACID guarantees are critical: users, orders, and cart.
  • Firestore — for the product catalog, which is read-heavy, semi-structured, and benefits from Firestore's flexible schema and global real-time replication.

Schema Overview:

-- Users table
CREATE TABLE users (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email         VARCHAR(255) UNIQUE NOT NULL,
    name          VARCHAR(255) NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    created_at    TIMESTAMPTZ DEFAULT NOW(),
    updated_at    TIMESTAMPTZ DEFAULT NOW()
);

-- Orders table
CREATE TABLE orders (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID REFERENCES users(id),
    status      VARCHAR(50) NOT NULL DEFAULT 'pending',
    total_cents INTEGER NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Order line items
CREATE TABLE order_items (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    order_id         UUID REFERENCES orders(id),
    product_id       VARCHAR(100) NOT NULL,  -- references Firestore doc ID
    quantity         INTEGER NOT NULL,
    unit_price_cents INTEGER NOT NULL
);

-- Cart
CREATE TABLE cart_items (
    id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id    UUID REFERENCES users(id),
    product_id VARCHAR(100) NOT NULL,
    quantity   INTEGER NOT NULL DEFAULT 1,
    added_at   TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE(user_id, product_id)
);

8.2 Secret Management

All secrets are stored in Secret Manager and injected into Cloud Run at runtime as environment variables. No secrets are stored in source code, Docker images, or environment files.

# Create secrets
echo -n "your-db-password" | gcloud secrets create nordlys-db-password \
  --data-file=- --replication-policy=automatic

echo -n "your-jwt-secret" | gcloud secrets create nordlys-jwt-secret \
  --data-file=- --replication-policy=automatic

# Grant Cloud Run service account access
gcloud secrets add-iam-policy-binding nordlys-db-password \
  --member="serviceAccount:nordlys-api-sa@nordlys-project.iam.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor"

8.3 Network Security

  • Cloud SQL is accessible only via private IP within data-subnet — no public IP assigned.
  • Cloud Run egress routes through the VPC Connector — database connections never traverse the public internet.
  • Cloud Armor WAF blocks OWASP Top 10 attack vectors (SQLi, XSS, RFI) before traffic reaches Cloud Run.
  • Load Balancer enforces HTTPS only — HTTP is redirected to HTTPS with a 301.

8.4 Application Security

  • JWT tokens used for user authentication — short-lived access tokens (15 min) + refresh tokens stored in Firestore.
  • Passwords hashed with bcrypt (cost factor 12) before storage in Cloud SQL.
  • CORS configured on Express to allow only the Nordlys Home frontend origin.
  • Rate limiting implemented at both Cloud Armor level (IP-based) and Express middleware level (user-based).
  • All Cloud Run requests require the Load Balancer as the ingress source — direct Cloud Run URLs are blocked.

11. Cost Optimisation

11.1 Cost Optimisation Strategies

  • Cloud Run scale-to-zero in dev and staging — instances shut down when idle, eliminating compute costs during off-hours.
  • Cloud SQL development tier (db-f1-micro) for dev — sufficient for development workloads at minimal cost.
  • Committed Use Discounts (CUD) on Cloud SQL production instance once baseline usage is established (typically after 3 months).
  • Cloud CDN reduces Cloud Storage egress costs by caching static assets at edge PoPs — particularly impactful for high-traffic scenarios.
  • Artifact Registry cleanup policy — automatically delete untagged container images older than 30 days to control registry storage costs.
  • Cloud Monitoring free tier — the first 150 MB of logs per project per month is free; implement log exclusion filters to avoid ingesting verbose debug logs in production.
  • Firestore free tier — Firestore's Spark plan free tier (50,000 reads/day, 20,000 writes/day) covers development usage entirely.
